## メモリに書くもの（コードから読み取れない情報のみ）

- 却下した選択肢とその理由（例: 「旧EAVパターンは不採用、複雑すぎる」）
- 運用ルール・ビジネスロジックの前提（例: 「未回答=不参加扱い」）
- スコープ判断（例: 「団体戦は後日追加」）
- ユーザーのフィードバック・指摘（例: 「ついでリファクタ禁止」）

## メモリに書かないもの（コードやgit logから導出可能）

- テーブル名・カラム名・型定義 → スキーマを読めばわかる
- APIエンドポイントの一覧 → routes/を読めばわかる
- ファイルパス・ディレクトリ構成 → codebaseを読めばわかる
- 実装の詳細手順 → git diffを読めばわかる

**Why:** メモリは膨れ上がると逆にノイズになる。コードから5秒で確認できることを二重管理する意味はない。2026-04-16にPhase 1-4の実装詳細（スキーマのカラム名等）をメモリに書いて「膨れ上がってもいいんですか」と指摘された。

## セッション終了時の同期手順

1. そのセッションで確定した**判断理由・運用ルール**をローカルmemoryに追記（実装詳細は書かない）
2. ローカルmemory → .claude/memory/ へコピー
3. worklog.md に作業記録を追記
4. git add → commit → push
5. 「メモリに書いた内容はコードから導出不可能か？」を自問してから終了報告

**How to apply:** 毎セッション終了時、および新しい設計判断をメモリに書く前に確認する。
