# kagetra_new 開発者ルールブック

このプロジェクトを進める上で、開発者（あなた）が従うべきルールとワークフロー。

---

## セッションの始め方

1. Claudeに現在の状況を確認させる（「今日何やるんだっけ」等でOK）
2. Claudeが自動で: git pull → memory同期 → worklog.md確認 → 前回の続きを提示
3. 途中から再開する場合は「前回の続きから」と伝えるだけでOK
4. **家でも会社でも同じ手順**。リモートリポジトリ経由で全て同期される

## セッションの終わり方

1. **「今日はここまで」と必ず伝える**
2. Claudeが自動で: claude-mem記録 → docs/worklog.md追記 → memory同期 → コミット → git push
3. 黙ってセッションを閉じない（記録が残らず別マシンで復元できなくなる）

---

## 機能開発の進め方

### 必ず守る順番

```
1. /grill-me で仕様をすり合わせる
2. /claude-mem:make-plan で実装計画を作る
3. 計画を確認し「OK」と伝える（← これは計画への同意。まだ実装は始まらない）
4. /claude-mem:do で実装を開始する（← これが実装開始の合図）
5. 実装中は1PR=1機能を守る
6. PRのCodexレビュー結果を確認し、対応/非対応を判断する
7. マージ前にスマホ実機で動作確認する
```

### やってはいけないこと

- **仕様確認なしで「これ作って」と言わない** — 必ず /grill-me を通す
- **計画なしで実装を始めない** — 必ず make-plan → 承認を経る
- **「ついでにこれも直して」と言わない** — 別PRで依頼する
- **Phase外の機能を今のPhaseに混ぜない** — 要望として伝え、該当Phaseで対応
- **Codexレビューを無視してマージしない** — 非対応にする場合は理由を残す
- **テストが落ちている状態でマージを指示しない**

---

## 並行作業

2つの機能を同時に進めたい場合は、**別のClaude Codeセッションを開くだけ**でOK。

- あなたが守ること: **1セッション = 1機能**。それだけ
- Claudeが自動でやること:
  - Git worktreeの作成・削除
  - 他ブランチとの競合チェック
  - DBマイグレーション番号の衝突回避
  - shared/パッケージの変更が安全か判断
  - マージ時のリベース・コンフリクト解消
  - 危険な並行作業の警告

## Claudeへの指示の出し方

### 使うコマンド

| 場面 | コマンド | 説明 |
|------|---------|------|
| 新機能の仕様検討 | `/grill-me` | 仕様を徹底的にすり合わせる |
| 要件定義+実装計画 | `/define-feature` | 要件定義書+実装計画+Issue作成 |
| 既存機能の改修計画 | `/fix-feature` | 監査結果に基づく改修要件定義 |
| 実装 | `/implement` | 1タスクずつworktreeで実装 |
| PR作成 | `/prepare-pr` | 実装完了後にPR作成→/reviewへ |
| レビュー依頼 | `/review` | Codex(VS Code)用レビュープロンプト生成 |
| レビュー指摘修正 | `/fix` | 指摘に基づくコード修正→再レビュー |
| マージ+出荷 | `/ship` | マージ+cleanup+memory同期+push |
| 小さなバグ修正 | `/quickfix` | 調査→修正→PR→レビューを一気通貫 |
| 大きなバグ対応 | `/bug-report` | Issue作成→調査→修正→PR |
| 既存機能の監査 | `/audit-feature` | ロジック・設計の監査レポート生成 |
| 作業記録 | `/history` | 1日の作業をログに残す |
| 過去の作業を検索 | `/claude-mem:mem-search` | 前に決めたこと・やったことを探す |

### 曖昧な指示を避ける

- NG: 「いい感じにして」「適当に直して」
- OK: 「イベント一覧画面で、終了済みイベントが非表示になるようにして」

### 設計変更は明示する

- 「ルール○○を変更したい」「この設計判断を見直したい」と明確に伝える
- Claudeが勝手に設計を変えることはないが、あなたから変更を依頼する際は理由も添える

## Phase管理

### 現在のPhaseを常に意識する

- Phase 1 → 2 → 3 → 4 の順番を飛ばさない
- 次のPhaseに進む前に品質ゲートを通す（Claudeが確認する）

### 品質ゲート時にやること

Claudeから「Phase N完了、次に進めますか？」と聞かれたら：
1. 本番環境でひととおり触って確認する
2. 問題なければ「次のPhaseへ」と伝える
3. 問題があれば具体的に報告する

---

## コードレビュー（Codex）

PRごとにCodex（VS Code拡張）でレビューを行う。追加費用なし（ChatGPTサブスク内）。

1. ClaudeがPRを作成し、**レビュー用プロンプト**を生成する
2. あなたがVS CodeでCodexを開き、そのプロンプトを渡す
3. Codexのレビュー結果を確認する
4. Claudeに対応を指示する
   - 「これ対応して」→ Claudeが修正
   - 「対応不要」→ そのままマージ

---

## やっていいこと

- いつでも仕様を変えたいと言っていい（ただし /grill-me を通す）
- 「この機能いらない」と途中で言っていい
- 「ここのUI気に入らない」と何度でもフィードバックしていい
- 「よくわからない」と言っていい — Claudeが説明する
- 作業の優先順位を変えたいと言っていい

---

## トラブル時

- 本番で問題が起きたら「本番で○○が起きている」と伝える
- Claudeが原因調査→修正 or ロールバックを提案する
- **自分で本番DBを直接操作しない** — 必ずClaude経由でマイグレーションを通す
